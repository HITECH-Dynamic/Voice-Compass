## Exp041A Results — ANV Dataset Index Strategy Inspection

### Objective

Determine whether transcript metadata contains sufficient information to locate audio files without scanning every Parquet shard.

### Findings

The answer is **no**.

Across all ANV repositories:

- Transcript CSVs contain `mediaPathId` references (for example, `VOICE_COLLECTION/...wav` or `PROMPT_COLLECTION/...wav`).
- Audio is distributed across many Parquet shards for each split.
- No transcript field identifies which Parquet shard contains a given recording.

Although repository inspection reveals how many Parquet shards exist for each split, there is no deterministic filename → shard mapping available from transcript or metadata files.

### Conclusion

Searching Parquet shards on demand is inefficient and does not scale.

The correct long-term solution is to build a persistent global lookup table:

mediaPathId
        ↓
Parquet shard

Once built, future preprocessing and training can resolve audio directly instead of scanning many large Parquet files.

### Outcome

✅ Exp041A completed successfully.

Research question answered.

## Experiment 041B — Global ANV Audio Index Generation

### Objective

Build the first production-ready filename → Parquet lookup index for all ANV datasets.

Unlike Exp040A, which validated the index format on a small bounded subset, Exp041B focuses on constructing a reusable global index that can support future preprocessing and multilingual training.

### Approach

- Process ANV audio Parquet shards incrementally.
- Record every audio filename together with its corresponding Parquet shard.
- Store the lookup as a reusable index.
- Allow indexing to resume incrementally rather than requiring one massive uninterrupted run.

### Expected Outputs

- data/afrivoices/indexes/anv_audio_index.parquet
- reports/dataset_analysis/anv_audio_index_summary.csv

### Success Criteria

- Incremental indexing supported.
- Previously indexed shards are not reprocessed.
- Output can be resumed after Colab disconnects.
- Filename lookup resolves directly to a Parquet shard.


### Design Revision — Resumable Global Audio Index

Based on the Exp040A runtime experience and the Exp041A inspection results, the production ANV audio index builder will support resumable execution.

Rather than requiring a single uninterrupted indexing run across every Parquet shard, the builder will:

- Detect previously indexed shards.
- Skip completed shards automatically.
- Append newly indexed filename → Parquet mappings.
- Save progress after every processed shard.
- Resume indexing after Colab disconnects or runtime limits.

### Why

The complete ANV corpus spans hundreds of large Parquet shards.

A resumable indexing strategy:

- prevents loss of work,
- avoids reprocessing completed shards,
- reduces Colab runtime waste,
- produces a reusable permanent infrastructure artifact.

This design becomes the production indexing architecture for Voice-Compass.
